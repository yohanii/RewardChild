# Frontend

Dart 러닝커브 때문에 RN 선택

## ⚔️ React Native vs Flutter — “우리 아이 보상체계” 기준으로 비교

| 구분 | **React Native** | **Flutter** |
| --- | --- | --- |
| **언어** | JavaScript / TypeScript | Dart |
| **개발 속도 (MVP)** | ✅ 빠름 — 웹/프론트 경험 있으면 바로 시작 가능 | ⛔ 러닝커브 있음 (Dart 익숙해야 함) |
| **UI 완성도** | 웹과 유사한 UI 빠르게 제작 가능 | ✅ 일관된 디자인, 애니메이션 매끄러움 |
| **성능 (프레임률)** | 중간 수준 (JS bridge overhead 있음) | ✅ 네이티브에 가까운 속도 |
| **패키지 생태계** | ✅ 방대함, 커뮤니티 크다 | 꾸준히 성장 중, 안정성도 향상 |
| **빌드 용이성 (iOS/Android 동시 배포)** | ✅ 쉬움, Expo로 MVP 배포 빠름 | ⛔ iOS 인증서 등 세팅 조금 복잡 |
| **애니메이션/UI 표현력** | 기본 수준, Lottie 등 추가 필요 | ✅ 강력 (CustomPainter, AnimationController 등 내장) |
| **웹 확장성 (PWA, Admin Web)** | ✅ React 기반 재활용 가능 | ❌ Flutter Web은 MVP엔 무거움 |
| **결제 연동 (Toss, IAP)** | ✅ React Native용 SDK 다양 | 가능하지만 SDK 다양성은 덜함 |
| **AI/Backend 연동 (FastAPI, Spring)** | ✅ Axios, Fetch로 손쉽게 연동 | OK, dio/http 패키지로 가능 |
| **장기 유지보수** | JS 기반이라 인력 구하기 쉬움 | Flutter 개발자 수는 아직 적음 |
| **빌드 크기** | 작음 (~30MB) | 큼 (~60MB 이상) |

## 🧩 1️⃣ Supabase (PostgreSQL 기반 BaaS)

> ✅ 현재 기준 최선의 “빠른 MVP + 저비용” 솔루션
> 

### 장점

- SQL 기반 → 구조화된 관계형 모델 (의뢰-보상-회원 구조에 적합)
- 인증(Auth), 스토리지, DB, API 자동 제공
- React Native SDK 완벽 지원 (`@supabase/supabase-js`)
- RLS(Row-Level Security)로 부모/자녀 구분 가능
- 무료 tier 매우 넉넉
- PostgreSQL이라 나중에 Spring Boot로 이전도 쉬움

### 단점

- Cloud Function 기능이 Firebase보다 약함
- 대규모 트래픽 대응 시 수동 최적화 필요
- 무료 tier는 월 요청수 제한 (50k) 있음

**👉 요약:** MVP엔 최고, 장기 확장 시 API 서버 따로 붙이면 끝.

---

## 🧩 2️⃣ Firebase (Google)

> 🔥 RN 연동 쉬움 / 푸시, 로그인 통합 강함 / 하지만 구조가 비SQL
> 

### 장점

- RN SDK 완성도 최고
- Google, Apple, Kakao 로그인 손쉬움
- Cloud Messaging(푸시), Analytics 기본 포함
- 서버 없이 클라이언트에서 거의 모든 기능 가능

### 단점

- NoSQL (문서형 DB) → 구조적 관계 (의뢰, 상점, 구매로그 등)에 비효율
- Cloud Function 사용 시 비용 급등
- 관리형 쿼리 한계 (JOIN 불가, 중첩문서 구조 복잡)
- “아이별, 부모별 통계” 같은 집계 쿼리 어렵다

**👉 요약:** 단순 채팅/To-do앱엔 완벽,

Yohan의 서비스처럼 “의뢰 ↔ 부모 ↔ 아이” 관계형 구조엔 **비추천**.

## 🧩 3️⃣ Render / Railway + Spring Boot or FastAPI

> 🧠 완전 자율형 서버 (클래식 백엔드)
> 

### 장점

- 완전한 자유 — 비즈니스 로직, 캐싱, 결제 연동, AI 기능 모두 구현 가능
- SQL 구조 마음대로
- 추후 확장에 유리 (결제, AI, 리포트 등)
- FastAPI는 가볍고 무료 배포 쉬움 (Railway 무료 tier 가능)

### 단점

- 초기 세팅/배포 시간 많음
- DB 관리 직접 해야 함 (RDS or Railway Postgres)
- 인증, 파일 업로드 직접 구현해야 함
- Cold start(대기시간) 존재 → 사용자 첫 진입 느릴 수 있음

**👉 요약:**

MVP 속도 < 비용 효율 < 확장성 우선이라면 나쁘지 않지만,

**“출시까지 2주 안에 빠르게 MVP 올리는 목표”라면 과함.**

---

## 🧩 4️⃣ PocketBase / Appwrite (자체 호스팅형 BaaS)

> 🧩 오픈소스 BaaS — 서버비는 거의 0원, 하지만 관리 필요
> 

| 항목 | PocketBase | Appwrite |
| --- | --- | --- |
| 언어 | Go (단일 실행파일) | PHP + Docker |
| 특징 | 초경량, 파일 하나로 서버 구동 | 기능 풍부 (Auth, DB, Storage) |
| 장점 | ✅ 무료, 초경량, 로컬 테스트 용이 | ✅ Firebase 대체 수준 완성도 |
| 단점 | ❌ 확장성 낮음, 고가용성 어렵다 | ❌ 세팅 복잡, Docker 필요 |

**👉 요약:**

- 단기 MVP엔 괜찮지만, 모바일 앱 스토어 등록 후 지속 운영엔 관리 부담 큼.
- 특히 부모-아이 관계/결제 로직을 추가하면 코드 서버화는 피하기 어려움.
